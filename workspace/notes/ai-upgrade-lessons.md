## 2026-03-04：独立机器人 vs 中心化调度

### 触发点
阅读文章《我用 OpenClaw 搞了家 16 人的公司：全员 AI，24 小时无休！》，发现其架构与当前方案有核心差异。

### 核心差异

| 维度 | 文章模式（16 个独立飞书机器人） | 当前模式（1 个机器人 + 内部路由） |
|------|---------------------------|----------------------------------|
| 身份隔离 | ✅ 每个机器人独立身份 | ❌ 所有回复都是"18 号" |
| 触发器独立 | ✅ 每个机器人有自己的 cron/触发器 | ❌ 所有触发器走同一个心跳 |
| 权限隔离 | ✅ 机器人 A 宕机不影响 B | ❌ 单点故障风险 |
| 运维复杂度 | ❌ 16 套配置、16 个 API Key | ✅ 只维护 1 个飞书应用 |
| 成本 | ❌ 每个机器人独立计费 | ✅ 统一计费，可全局优化 |
| 用户体验 | ✅ @财务机器人得财务回答 | ❌ 用户需要知道"18 号能处理所有事" |

### 架构本质对比

**文章模式 = 去中心化架构**
- 每个 AI 员工是独立进程，有自己的"命"
- 更像真实公司：财务部不会等技术部心跳

**当前模式 = 中心化调度**
- 18 号是总控，sub-agent 只是临时任务
- 更像"超级助理 + 外包团队"

### 混合架构方案（待实施）

```
1 个主机器人（18 号） + N 个独立专家机器人

主机器人职责：
- 处理日常对话、路由决策
- 维护全局记忆、心跳、日志
- 协调跨专家协同任务

独立机器人（只给核心专家）：
- 财务助手：每天固定报表，独立触发器
- 技术顾问：独立监控、独立告警
- 数学老师：已在独立 workspace，保持现状
```

**选择标准：**
- 高频、独立、边界清晰的场景 → 拆成独立机器人
- 需要协同、共享上下文的场景 → 保持 sub-agent 模式

### 待决策问题

1. **飞书应用管理**
   - 是否需要为每个独立机器人创建独立飞书应用？
   - 还是用同一个应用的不同 Bot 身份？

2. **记忆共享机制**
   - 独立机器人如何访问全局 MEMORY.md？
   - 是否需要只读共享 + 独立笔记目录？

3. **触发器去重**
   - 主机器人和独立机器人的心跳如何避免重复执行？
   - 是否需要统一触发器注册表？

4. **成本核算**
   - 独立机器人的 API 调用如何单独计量？
   - 是否需要为每个机器人配置独立模型策略？

## 2026-03-04：心跳机制增强

### 触发点
从文章中学到动态心跳和 stale 任务恢复机制。

### 已实施改进

1. **动态心跳频率**
   - 有活跃 sub-agent → 心跳从 30m 切到 5m
   - 无活跃任务 → 恢复 30m

2. **Stale 任务自动恢复**
   - 30 分钟无进展的任务自动唤醒
   - 记录恢复动作到 `heartbeat-state.json`

### 待验证
- 下次心跳实际跑一遍检查逻辑
- 确认 stale 检测能正确触发
