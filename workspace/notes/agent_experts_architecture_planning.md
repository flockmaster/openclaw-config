# OpenClaw 智能体能力强化与“自动请专家”架构规划

## 一、 为什么“自动请专家”的思路是对的？

赋予智能体“抓抖音、读网页、做搜索”能力，解决了“输入（感知层）”的问题。但面对大量且复杂的专业信息时，单个通用大模型往往处理不过来，容易产生幻觉或回答不够深入。

“自动请专家来解决专业化问题”在 AI 领域被称为 **多智能体协作系统 (Multi-Agent System, MAS)** 或者 **Agent 路由架构 (Agent Routing / Mixture of Experts)**。这正是实现复杂业务落地的必经之路：

1. **突破单体模型的性能天花板**：一个包含了所有 prompt 和所有工具的超级 Agent 会非常臃肿，容易在工具间迷失。将任务拆分给特定领域的“专家”（拥有专属 Prompt 和小工具集），成功率会成倍提升。
2. **职责解耦**：前端问题路由给“前端研发 Agent”，数据统计路由给“数据分析 Agent”。各自独立，维护更方便。
3. **人机协同 (Human-in-the-loop)**：“专家”不一定全是 AI！遇到危险或超出 AI 能力边界的操作，AI 可以自动在飞书群 `@人类专家` 寻求审批或帮助，构建企业级防火墙。
4. **杜绝质量不可控 (Verified Skills vs. Hallucinated Prompts)**：临时让大模型自动生成一段“你是产品经理”的提示词，效果是未经测试且极不稳定的。真正的“请专家”应该是去**已验证的 Skill/Agent 技能仓库（如本地成熟模板库或内部市场）**中拉取/复制经过长期调优的、配备专属工具栈的成熟 Agent。这确保了请来的是“行业大拿”，而非街边随便拉来的路人。

## 二、 如何规划与实现“自动请专家”功能？

在现有 OpenClaw 架构下（考虑到已有的飞书 Gateway 路由），可按以下 4 个阶段落地：

### Phase 1: 建立“专家注册表”与“技能拉取机制” (Skill Hub & Registry)
在架构设计上，我们需要一个类似 `skill_market` 的机制。当 Manager 需要特定专家时，不再是“当场写 prompt 创建”，而是去预设的成熟仓库中**拉取并复制完整的 Agent 配置**（包含充分打磨的 system_prompt、专有 tools 脚本、甚至工作流 workflows）。

专家档案配置（如 `openclaw.json`）示例：
* **AI 专家**：从成熟仓库 (Skill Repo) 动态加载的验证级模板。
* **人类专家**：特定领域负责人的飞书 Open ID。

```json
"experts": {
  "product_manager": {
    "type": "ai",
    "source": "skill_hub://product_manager_pro_v2", 
    "description": "经过大量真实项目验证的产品经理 Agent，内置画原型、PRD拆解工具",
    "status": "ready"
  },
  "dba_expert": {
    "type": "human",
    "feishu_id": "ou_xxxxx12345",
    "description": "数据库管理员，处理删库、数据查询等高危操作"
  }
}
```

### Phase 2: 实现智能路由中枢与多源猎头系统 (Manager & Expert-Scout)
主 Agent 升级为“项目经理 (Manager)”角色（如由 Gemini 3.1 Pro 这样的高智力模型驱动）：
1. **意图识别与静态路由**：Manager 阅读用户需求，判断是否可以直接回答，或者本地 `expert-registry.md` 中是否已有对口的专家。如果有，直接通过 `@expert-id` 委派（如飞书群绑定）。
2. **动态多源发现 (expert-scout)**：若本地无对口专家，Manager 不会自己“硬编”提示词，而是触发一个 `expert-scout` (专家猎头) Skill。
3. **多途径搜索与择优录取**：`expert-scout` 会并发去多个外部成熟生态（如 MiniMax Agent 平台、GitHub Awesome Prompts、LangChain Hub 或垂直开源项目）搜寻候选专家。
4. **本地高级智力评估与标准化化**：Manager 拿到多源回传的候选专家配置后，进行交叉对比和打分评估。选出最优解后，对其进行“本地化改造”（生成符合 OpenClaw 原生规范的 `SOUL.md` 和带有血统记录的 `EXPERT-META.md`），将其正式注册为本地可控的真专家。

### Phase 3: 上下文交接与异步执行
* **AI 专家**：OpenClaw 后台拉起子专家会话，传入上下文。子专家自行调用工具得出结论，返回给 Manager。
* **人类专家**：OpenClaw 在飞书发消息：“@DBA专家，这里有一个获取生产环境订单数据的需求，我已准备好 SQL，请确认是否执行？[附带上下文卡片]”。

### Phase 4: 结果综合与交付 (Synthesis)
Manager 拿到“专家”回复后，不生硬转发，而是进行融会贯通，用友好统一的口吻输出最终答案。

## 三、 为了让 OpenClaw 更智能，还应该增加哪些能力？

除了“外部知识获取”和“专家协作”，建议补充以下能力系统（按优先级排序）：

### 1. 代码沙盒与执行力 (Code Interpreter / Sandbox) - **强烈推荐**
获取知识后，AI 需要能“动手”。
提供安全的本地 Docker 沙盒或受限环境，让 AI 可以**自己运行 Python、Shell 代码**。
* **场景**：抓取视频数据后，AI 自动写 Python 脚本数据清洗，生成 Excel 图表，再通过飞书发送。

### 2. 长记忆与经验库 (Long-term Memory / RAG)
突破会话记忆限制，构建经验库。
* **能力实现**：引入轻量向量数据库 (如 ChromaDB) 或本地索引。解决完复杂任务后，自动生成“总结”存入经验库。
* **场景**：下次遇到类似报错，Agent 能回忆起“上个月处理过这个跨域问题，当时是因为 NGINX 配置...”。

### 3. 企业系统深度集成 (Enterprise API Hub)
打通更多工作纽带：
* **GitLab/GitHub**：查阅代码、自动提 PR、总结 Code Review。
* **Jira/Tapd**：自动生成工单，查询 Bug 进度。
* **飞书云文档**：调研后直接自动生成排版精美的云文档报告。

### 4. 自我反思与错误纠正循环 (Self-Reflection Loop)
实现“重试与自我纠正”。
* **机制**：调用工具 -> 捕获异常 -> 分析错误日志 -> 调整参数/换方案 -> 再次重试，直到成功或达上限。

## 四、 标准化开发流程与质量保障体系 (Skeleton-Driven TDD Workflow)

> **核心洞察**：主 Agent 不能随意分配任务。必须建立在标准化开发流程之上，确保每个环节有明确的输入、输出和质量门禁。

### 1. 完整流程概览


### 2. 骨架驱动 TDD 的核心设计（Skeleton-Driven TDD）

#### 为什么选择骨架驱动？

在多智能体 TDD 流程中，最大的挑战是：**QA Agent 和 Dev Agent 拥有完全隔离的上下文，如果不事先约定接口契约，两者写出的代码根本对不上号。**

我们评估了三个方案：

| 方案 | 描述 | 评估 |
|---|---|---|
| **方案 1: 接口声明先行** | 主 Agent 或架构 Agent 先输出强类型接口声明 | ✅ 可行，但增加主 Agent 工作量 |
| **方案 2: 骨架代码驱动** ⭐ | Dev 先输出空骨架（只有签名无逻辑），QA 据此写测试 | ✅ **最终选定**，最小成本、最务实 |
| **方案 3: BDD 自然语言测试** | 用 Given-When-Then 自然语言描述测试场景 | ❌ **否决**，见下方分析 |

#### 方案 3 (BDD) 被否决的原因

> ⚠️ **关键教训**：自然语言不能在机器上直接运行。

- 如果 QA 只用自然语言"审查"代码而不执行物理测试，本质上是让大模型"闭眼盖章"，极易产生幻觉
- 要让自然语言测试真正可运行（如 Cucumber/Behave），需额外编写 Step Definitions 代码，Token 消耗反而翻倍
- 增加"自然语言到代码"的转换层，出错率随之翻倍

#### 骨架代码示例

```python
# skeleton: calculator.py（骨架文件 — Dev Agent 第一步输出）
from abc import ABC, abstractmethod

class Calculator(ABC):
    """计算器服务接口"""

    @abstractmethod
    def add(self, a: float, b: float) -> float:
        """两数相加"""
        ...

    @abstractmethod
    def divide(self, a: float, b: float) -> float:
        """两数相除，除数为零时抛出 ValueError"""
        ...
```

```python
# test_calculator.py（QA Agent 基于骨架编写 — 此时全部为红）
import pytest
from calculator import Calculator

class ConcreteCalculator(Calculator):
    """QA 用最小实现来测试接口契约"""
    def add(self, a, b): return a + b
    def divide(self, a, b):
        if b == 0: raise ValueError("除数不能为零")
        return a / b

def test_add_normal():
    calc = ConcreteCalculator()
    assert calc.add(1, 2) == 3

def test_divide_by_zero():
    calc = ConcreteCalculator()
    with pytest.raises(ValueError):
        calc.divide(1, 0)
```

### 3. 质量闭环防护机制

为防止"局部测试通过、整体集成崩溃"的风险：

1. **物理测试优先**：所有测试必须能在系统中真正运行并产生红/绿结果，拒绝任何口头确认的"虚拟测试"
2. **接口契约不可篡改**：骨架代码一旦确定，Dev 不得擅自修改函数签名和类结构，如需变更必须走变更流程
3. **主 Agent 校验规则**：
   - 检查测试是否全部通过（读取测试运行结果，非口头询问）
   - 检查代码是否符合骨架定义的接口契约
   - 在规划文档中更新任务状态
4. **集成 Agent（可选）**：当项目规模较大时，增设独立的集成 Agent 负责多模块拼装和集成测试

### 4. 上下文窗口优化策略

本架构的根本驱动力是**突破大模型上下文窗口限制**：

- **主 Agent**：只负责协调调度和苏格拉底式需求澄清，上下文保持精简
- **子智能体**：每个只聚焦单一原子任务，上下文干净纯粹，注意力高度集中
- **知识传递**：通过文件（PRD、规划文档、骨架代码）而非上下文传递信息，避免信息在传话中丢失
- **记忆固化**：任务完成后的关键结论写入持久化文件（而非仅存在于对话上下文中）
